Concept 01

What Is a Token?

Models don't read letters or whole words — they read tokens: sub-word chunks produced by a tokenizer before any prediction happens.

  • A tokenizer splits text into a fixed vocabulary of common chunks — whole words, word pieces, punctuation, even single characters.
  • Splitting below the word level lets the model handle rare words, typos, and new languages without an infinite vocabulary.
  • Common short words are often one token; longer or unusual words split into two or more.
  • Roughly speaking, 1,000 tokens ≈ 750 English words — but code, non-English text, and numbers tokenize less efficiently.

Concept 02

From Probabilities to a Chosen Word

For every next token, the model doesn't output one answer — it outputs a full probability ranking over its entire vocabulary.

6%
glitter

  • This ranked list is called the logits (turned into percentages via a function called softmax).
  • A sampling strategy then picks one token from this distribution — it doesn't always take the top choice.
  • Two dials control that choice: temperature and top-p — covered next.
Concept 03

Temperature: the Randomness Dial

Temperature reshapes the probability ranking before a token is sampled — it doesn't change what the model "knows," only how boldly it gambles.

  • Low temperature (≈0–0.3): sharpens the distribution — the top choice dominates, output is focused and repeatable.
  • High temperature (≈1.0–2.0): flattens the distribution — lower-ranked tokens get a real chance, output is varied and surprising.
  • Good defaults: low temperature for factual Q&A or code; higher temperature for brainstorming or creative writing.
  • Temperature is a request parameter you set per API call — it isn't a property of the model itself.
temperature = 0.7
A balanced default: mostly coherent, with a little room for variety.
See It In Action

Same Prompt, Two Temperatures

Prompt: "Finish this sentence: The old lighthouse keeper opened the door and saw..."

temperature = 0.1 — deterministic
"...saw that the light was still working, just as it had every night for thirty years."

Run it five times → nearly the same sentence every time.
temperature = 1.6 — wild
"...saw a raccoon in a captain's hat, arguing with the fog about maritime law."

Run it five times → five wildly different, sometimes incoherent, scenes.

The lesson: temperature doesn't add "creativity" from nowhere — it changes how far the sampler strays from the model's most likely guess. Push it too high and fluency itself breaks down.

Concept 04

Top-p: a Different Kind of Cutoff

Top-p (nucleus sampling) restricts the candidate pool by cumulative probability, instead of reshaping the whole curve like temperature does.

How it works

The model keeps adding candidates — highest probability first — until their combined probability reaches p (e.g. 0.9), then samples only from that shortlist.

Why use both

Temperature and top-p are often combined: temperature sets "how sharp," top-p sets "how many options are even on the table."

  • top-p = 1.0 considers the whole vocabulary; top-p = 0.1 restricts sampling to a very narrow, high-confidence shortlist.
  • Most APIs let you set both — in practice, teams usually tune one and leave the other at its default.
Concept 05

The Context Window

A model can only "see" a fixed number of tokens at once — the system prompt, conversation history, and the model's own output all share this one budget.

  • Context windows range from a few thousand tokens to well over a hundred thousand, depending on the model.
  • Once the budget is full, the oldest turns must be dropped, summarized, or truncated — the model simply can't see them anymore.
  • A longer context window doesn't guarantee better use of it — models can still miss details buried in the middle of a huge prompt.

Concept 06

Cost & Latency: Tokens Are the Meter

API providers bill per token, usually with separate rates for input and output — and more tokens generally means a slower response too.

Prompt shapeApprox. input tokensEffect
Short question, no history~20Fast, cheap
+ 10-turn chat history~1,200Slower, more expensive
+ pasted document context~6,000Noticeably slower, priced accordingly
  • Trimming unnecessary history or pasted text is a direct lever on both cost and response time.
  • Longer requested outputs (e.g. "write 2,000 words") also add latency — generation is sequential, one token at a time.

Common Pitfalls

Tokens & Sampling Myths

"Temperature 0 is always identical"

It's close to deterministic, but ties and backend nondeterminism can still cause tiny variation between calls.

"Tokens = words"

Tokens are sub-word pieces — punctuation, whitespace, and word fragments each count too.

"More context is always better"

Larger context costs more, runs slower, and doesn't guarantee the model attends evenly to everything inside it.

"Higher temperature = smarter answers"

Higher temperature means more varied, not more correct — it can just as easily introduce errors.
