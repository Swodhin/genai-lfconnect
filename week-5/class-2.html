Concept 01

Plan — What Should I Do Next?

  • The model looks at the goal, the conversation so far, and the tools it's been given
  • It decides: answer directly, or call a tool first?
  • Example: "What's the weather in Kathmandu, and should I bring a jacket?"
  • The model plans to call get_weather before it can answer the jacket question

Concept 02

Tool / Function Calling — How It Works

  • You describe each tool with a name, a description, and a JSON-schema of its parameters
  • Those descriptions are sent to the model alongside the conversation
  • Instead of plain text, the model can return a structured tool call: a function name plus arguments
  • Your code executes the real function — the model never runs anything itself

Concept 03

Act — Executing the Call

The model's response requests a tool call, shaped like this:

{
  "name": "get_weather",
  "arguments": { "location": "Kathmandu" }
}

Your code reads this, calls get_weather("Kathmandu"), and gets a real result back.

Concept 04

Observe — Feeding Results Back

The tool's result is sent back as a new message, and the model continues:

TOOL. { "temp_c": 14, "condition": "rain" }

ASSISTANT. "It's 14°C and rainy in Kathmandu — bring a jacket."

Concept 05

Why Loop Instead of One Shot?

  • Models can't reach live or private data alone — tools ground answers in real, current information
  • Multi-step problems often need more than one observation before an answer is ready
  • Feedback lets the model correct course — e.g. retry a search with a better query
  • Without the loop, "using a tool" would just be one extra, disconnected step

Pitfalls & Misconceptions

Where This Breaks

Forgotten Result

Not sending the tool's result back — the model repeats itself or stalls.

Vague Descriptions

A fuzzy tool description leads the model to call the wrong tool, or none at all.

No Validation

Running a tool with unchecked arguments can crash your code or misbehave.

No Turn Limit

Without a cap, the model can loop far longer — and cost far more — than expected.

Recap

Key Takeaways

  • Tool calling lets the model request an action instead of just text
  • Your code always executes the real function, never the model
  • Results must be fed back as a new message for the loop to continue
  • Plan → Act → Observe repeats until the model has a final answer
  • Always design a stopping condition — don't let it loop unchecked
