What Is an Embedding, Really?

  • An embedding is a list of numbers — a vector — that represents the meaning of a piece of text.
  • Produced by a trained embedding model: text goes in, a vector comes out.
  • Texts with similar meaning end up with vectors that sit close together in that numeric space.
  • "Dog" and "puppy" land near each other. "Dog" and "stock market" land far apart.
  • Real embedding models use hundreds or thousands of dimensions — far more than we can draw, but the same idea.
  • Embeddings turn "does this mean the same thing?" into a math problem: how close are two points?

Vector Space: Meaning as Geometry

  • Picture every document and every query as a dot in space.
  • Related topics cluster together — the same way a library groups related books on a shelf.
  • A new question becomes its own dot the instant it's embedded.
  • The nearest dots are its nearest neighbors — the most relevant chunks to retrieve.
  • We can only draw 2 dimensions here; real models use hundreds. The geometry is identical.

Concept

Measuring Similarity With Cosine

  • To rank neighbors we need a number for "how close" two vectors are.
  • Cosine similarity measures the angle between two vectors — it asks "do they point the same direction?"
  • Scores run roughly from -1 (opposite) to 1 (identical direction); most text embeddings land between 0 and 1.
  • Higher score = more similar meaning. Sort documents by score, keep the top few.
cosine_similarity(A, B) = dot(A, B) / (norm(A) * norm(B))

Semantic Search vs. Keyword Search

Keyword Search

  • Matches exact words in the text
  • "puppy food" misses an article titled "canine nutrition"
  • Fast, simple, and well understood
  • Struggles with synonyms and paraphrasing

Semantic Search

  • Matches meaning, not exact wording
  • "puppy food" correctly finds "canine nutrition"
  • Needs an embedding model plus vector comparison
  • Handles synonyms, paraphrase, and typos gracefully

Many production systems blend both — "hybrid search" — for the best of each.

Vector Databases: Where Embeddings Live

Chroma

  • Open-source, runs locally or embedded in your app
  • Great for prototyping and smaller projects
  • Simple Python API, minimal setup

FAISS

  • Meta's similarity-search library
  • In-memory and extremely fast at scale
  • No server — you manage persistence yourself

Pinecone

  • Fully-managed cloud vector database
  • Scales to billions of vectors; ops handled for you
  • Good fit for production teams

All three do the same core job: store vectors, and find nearest neighbors fast.

Common Pitfalls

Misconceptions Worth Correcting

Chunking Affects Quality

Chunks that are too big or too small hurt embedding matches — more on this next class.

Use the Same Model on Both Sides

Embed your documents and your queries with the same model, or distances become meaningless.

Cosine Isn't Magic

An irrelevant document can still score deceptively high — always sanity-check top results.

Cost Scales With Volume

Every chunk you embed costs time and money — plan for it at scale.

Recap

Key Takeaways

  • An embedding is a vector that represents meaning; nearby vectors mean similar meaning.
  • Cosine similarity gives us a number for "how close" two vectors point.
  • Semantic search matches meaning; keyword search matches literal words.
  • Vector databases like Chroma, FAISS, and Pinecone store and search vectors at scale.
  • Nearest-neighbor search is the retrieval half of RAG — generation is next.
