>>> concept 01

Client, Server, Request, Response

  • An API is a contract: send a request in an expected shape, get a response back
  • HTTP is the protocol underneath — your script is the "client," the AI provider runs the "server"
  • Every request has a method (GET to fetch, POST to send data), a URL (the endpoint), headers (metadata like auth), and often a body (the payload)
  • Every response has a status code (200 OK, 401 unauthorized, 429 rate limited, 500 server error) and a body
  • Calling an LLM API is just: POST a JSON body to an endpoint, get a JSON body back

Secrets & Environment Variables

  • API keys are secrets — anyone with your key can spend your money and act as you
  • Never hardcode a key in a script, and never commit one to git
  • Environment variables keep secrets outside your code — set them in your shell or a .env file, read with os.environ
  • os.environ.get(...) returns None — not an error — if a variable isn't set; always check before using it
import os api_key = os.environ.get("OPENAI_API_KEY") if not api_key: print("Missing OPENAI_API_KEY — set it first.")

The requests Library Basics

  • requests is Python's standard tool for HTTP calls — pip install requests
  • requests.get(url) fetches data; requests.post(url, json=payload) sends a JSON body
  • response.status_code tells you if it worked; response.json() parses the reply
  • Always set a timeout= so a hung connection doesn't freeze your script
import requests resp = requests.get( "https://api.example.com/ping", timeout=10 ) print(resp.status_code, resp.json())

>>> concept 04

Anatomy of a Chat Completion Request

  • Endpoint: a URL like the OpenAI or Anthropic chat/messages endpoint
  • Headers carry your API key and content type
  • Body carries the model name, a list of role-tagged messages, and settings like temperature
  • The reply nests the model's text inside the response body — the SDK unwraps it for you
{ "model": "gpt-4o-mini", "messages": [ {"role": "system", "content": "You are concise."}, {"role": "user", "content": "What is a variable?"} ] }

Reading the Response, Safely

  • Wrap the call in try/except — network calls fail in ways your code can't prevent
  • Check the status before trusting the data — a 429 or 401 needs different handling than a 200
  • Set a max_tokens limit so a runaway response doesn't blow your budget
  • Log the raw response while developing — you can't debug what you can't see
try: resp = client.chat.completions.create( model="gpt-4o-mini", messages=messages, max_tokens=200 ) text = resp.choices[0].message.content except Exception as e: text = f"Request failed: {e}"

>>> common pitfalls

Where Beginners Get Tripped Up

Hardcoding keys

Leaking a key in a script — or a public repo — is mistake #1. Use env vars and add .env to .gitignore.

Ignoring status codes

A 200, a 429 (rate limited), and a 401 (bad key) all need different handling — check before parsing.

No timeout

A hung request can block your script forever. Always pass timeout=.

Confusing roles

system sets behavior once, user is the human turn, assistant is the model's turn.
