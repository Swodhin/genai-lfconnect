Concept

Free Text Doesn't Scale

The moment a prompt's output feeds into more code — a database, a UI, another API call — free-form prose becomes a liability.

  • Structured output (JSON, CSV, XML) can be parsed deterministically; prose has to be re-interpreted every time.
  • Downstream systems need consistent field names and types — not "sometimes a list, sometimes a paragraph."
  • Structured output is what makes chaining prompts, function calling, and agent tool use possible.
  • It's still a prompting problem first — you must specify the exact shape you want.

Ask for Exactly What You Want

Weak draft Give me the product details as JSON.↖ which fields? which types?
Revision Return ONLY valid JSON, no other text, matching exactly this shape: {"name": string, "price_usd": number, "in_stock": boolean}.
  • Show the exact keys, types, and an example — don't just say "as JSON."
  • Say "return ONLY JSON, no explanation, no markdown code fences" to stop extra prose.

Concept

Letting the API Enforce the Shape

Modern LLM APIs offer structured-output modes — like JSON mode or a strict JSON schema — so the model's raw output is guaranteed parseable.

  • OpenAI: response_format={"type": "json_schema", ...} constrains output to your schema at the token level.
  • Anthropic: describing the schema in the system prompt plus a tool/function definition achieves the same reliability.
  • Schema constraints reduce, but don't eliminate, the need for validation — types can still be logically wrong (a price of −5).
  • Prefer schema mode over "please output JSON" whenever the API supports it — it's more reliable, not just more polite.

Concept

Trust, But Verify

Never pass model output straight into production code — validate it first, the same way you'd validate user input.

  • Wrap json.loads() in a try/except — catch malformed JSON before it crashes your pipeline.
  • Check required keys are present and types match what you expect, or use a schema library like pydantic.
  • On failure, retry with a corrective follow-up: "Your last response wasn't valid JSON. Return ONLY the corrected JSON."
  • Log failures — patterns in what breaks tell you what to fix in the prompt, not just the code.

Concept

A Process for Fixing a Broken Prompt

  • Isolate — strip the prompt to its smallest form that still fails; remove unrelated instructions.
  • Reproduce — run it multiple times; is it consistent or intermittent (temperature-sensitive)?
  • Compare — change exactly one variable (wording, example, temperature) between attempts.
  • Localize — is it a formatting bug, a factual bug, or a misunderstanding of the task?
  • Patch — add the missing constraint, example, or delimiter: the smallest fix that addresses the cause.
  • Re-test — confirm the fix on several inputs, not just the one that first failed.

Concept

Failure Patterns You'll See Constantly

01Extra Prose

"Sure, here's the JSON:" before the object. Fix: say "output ONLY JSON, nothing else."

02Markdown Fences

Output wrapped in ```json ... ```. Fix: forbid markdown, or strip fences in code before parsing.

03Truncated Output

JSON cut off mid-object. Fix: raise the max token limit, or ask for a shorter, paginated result.

04Wrong Fields/Types

"Price" instead of "price", a string instead of a number. Fix: show an example with exact keys and types filled in.

05Hallucinated Fields

The model invents extra keys. Fix: say "do not include any keys other than those listed."

06Inconsistent Shape

An array sometimes, an object other times. Fix: pin the top-level shape with an example, or use schema mode.

Watch For This

Debugging Anti-Patterns

✕

Changing Everything at OnceYou can't tell which change fixed — or broke — it.

✕

Testing OnceOne lucky run doesn't prove reliability; temperature makes output non-deterministic.

✕

Ignoring TemperatureHigh temperature on a structured-output task invites inconsistency — lower it.

✕

Fixing the Code, Not the PromptA fragile regex "fix" just moves the bug downstream.

✕

No FallbackShipping without a retry-on-failure path guarantees production breakage eventually.

✕

Skipping Edge CasesOnly testing the happy path — never the empty, huge, or malformed input.

Recap

What We Revised

  • Structured output turns prose into something code can trust — and it starts with a precise prompt.
  • Always specify exact keys, types, and "JSON only" — then validate what comes back.
  • Prefer schema-constrained modes over "please output JSON" when your API supports them.
  • Debugging is a process: isolate, reproduce, compare one variable at a time, patch, re-test.
  • Most failures repeat a small set of patterns — extra prose, wrong keys, truncation, hallucinated fields.
